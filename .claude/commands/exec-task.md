# タスク実行コマンド

指定されたタスクファイルを読み込み、タスクを順番に実行します。

---

## 実行フロー

### Step 1: タスクファイルの読み込み

1. 指定されたタスクファイル（$ARGUMENTS）を読み込む
2. タスクファイルが存在しない場合は `tasks/` ディレクトリの一覧を表示
3. タスクの一覧と概要をユーザーに提示

### Step 2: タスクの実行

各タスクを以下のルールで実行してください：

#### 実装タスクの場合

**@fullstack-engineer に委譲してください。**

実装タスクの例：
- ファイルの作成・編集
- 機能の実装
- リファクタリング
- 設定ファイルの更新
- データベースのマイグレーション

```
@fullstack-engineer [タスクの内容を具体的に指示]
```

#### テストタスクの場合

**@qa-engineer に委譲してください。**

テストタスクの例：
- 単体テストの作成・実行
- 統合テストの作成・実行
- E2Eテストの作成・実行
- テストカバレッジの確認

```
@qa-engineer [タスクの内容を具体的に指示]
```

#### 画面確認を伴うテストの場合

**@qa-engineer に委譲し、Playwright MCP の使用を指示してください。**

画面確認が必要な例：
- UIの表示確認
- フォームの動作確認
- 画面遷移の確認
- レスポンシブデザインの確認
- スクリーンショットの取得

```
@qa-engineer Playwright MCP を使用して以下の画面テストを実施してください：
[テスト内容を具体的に指示]

使用するPlaywright MCPツール：
- playwright:browser_navigate - ページ遷移
- playwright:browser_screenshot - スクリーンショット取得
- playwright:browser_click - 要素のクリック
- playwright:browser_type - テキスト入力
- playwright:browser_snapshot - ページ状態の取得
```

### Step 3: タスク完了の確認

各タスク完了後：

1. **完了条件の確認**: タスクファイルに記載された完了条件を満たしているか確認
2. **テストの実行**: 関連するテストがあれば実行
3. **ユーザーへの報告**: 完了したタスクと結果をユーザーに報告
4. **次のタスクへ**: ユーザーの確認を得てから次のタスクに進む

---

## タスク判定基準

| キーワード | 担当 | 備考 |
|-----------|------|------|
| 実装、作成、追加、修正、更新 | @fullstack-engineer | コード変更を伴う作業 |
| テスト、検証、確認 | @qa-engineer | 品質確認作業 |
| 画面、UI、表示、ブラウザ | @qa-engineer + Playwright | 画面操作を伴うテスト |

---

## 実行時の注意事項

1. **順序を守る**
   - タスクは記載順に実行する
   - 依存関係がある場合は前のタスクの完了を確認してから進む

2. **エラー時の対応**
   - エラーが発生したら即座にユーザーに報告
   - 自動的にリトライせず、ユーザーの判断を仰ぐ

3. **スコープを守る**
   - タスクファイルに記載された範囲のみ実行
   - 追加の作業が必要な場合はユーザーに確認

4. **進捗の報告**
   - 各タスク開始時に「Task N.X を開始します」と報告
   - 各タスク完了時に結果を報告
