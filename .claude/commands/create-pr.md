# PR文作成コマンド

あなたはコードレビューに精通したシニアエンジニアです。
変更内容を分析し、レビュアーにとってわかりやすいPR文を作成してください。

---

## 実行手順

### Step 1: 差分の確認

以下のコマンドで変更内容を確認してください：

```bash
# ステージング済みの差分
git diff --cached --stat
git diff --cached

# まだステージングしていない差分がある場合
git diff --stat
git diff

# 直近のコミットとの差分（コミット済みの場合）
git log -1 --oneline
git diff HEAD~1 --stat
```

### Step 2: 変更内容の分析

差分から以下を把握してください：

1. **変更されたファイル一覧**
2. **変更の種類**（新規追加/修正/削除）
3. **主要な変更点**
4. **影響範囲**

### Step 3: PR文の作成

`.github/pull_request_template.md` の形式に従い、以下の内容で `PR.md` を作成してください：

```markdown
## 概要

[この変更で何を達成するか、なぜこの変更が必要かを簡潔に記述]

- 変更の目的
- 解決する課題やIssue番号（あれば）

## 設計

[どのように実装したかを記述]

### 変更ファイル

| ファイル | 変更内容 |
|---------|---------|
| `path/to/file` | [変更内容の説明] |

### アーキテクチャ/設計判断

[重要な設計判断があれば記述]

- なぜこのアプローチを選んだか
- 他に検討した選択肢（あれば）

## 参考文献

[参考にしたドキュメントやリソース]

- [リンクや資料名]

## 自分でテストした内容

[どのようにテストしたかを記述]

- [ ] [テスト項目1]
- [ ] [テスト項目2]

### テスト手順

1. [再現手順があれば記述]

## その他レビューアへの共有事項

[レビュー時に特に見てほしい点、懸念事項、議論したい点など]

- 特に見てほしい箇所
- 既知の問題や制限事項
- 今後の課題（あれば）
```

---

## 出力ルール

1. **出力先**: プロジェクトルートの `PR.md`
2. **上書き**: 既存の `PR.md` がある場合は完全に上書きする
3. **言語**: 日本語で記述する

---

## 注意事項

- 変更内容を正確に反映する（推測で書かない）
- レビュアーが理解しやすい粒度で記述する
- 技術的な詳細は「設計」セクションに集約する
- テスト内容は具体的に記述する
